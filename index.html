怎麼變成 GAS 自動化工作流？

從紙本到戰情室的進化之旅 🚀

真正的麻煩，不是「收到」...

  • 資料亂飛：
    電話 紙本 LINE Email Google評論
  • 這是一般建議，還是會爆炸的高風險客訴？
  • 寄出去之後，到底結案了沒？
  • 主管問：現在總共幾件？我們答得出來嗎？ 😱

我第一個不是想寫程式，
而是先問：這流程哪裡卡住？

  • 哪裡 一直在重複？
  • 資料 散在哪裡？
  • 有沒有 固定的判斷？
  • 發生事情時，要 通知誰？
  • 寄出去後，怎麼追蹤？
  • 主管最後 想看什麼？

Q1：哪裡一直重複？

每次都在問家屬一樣的基本資料。

用 Google Form 統一入口

⚙️ GAS 出場點：表單送出後，自動開始處理 (onFormSubmit)

(💡 現場 Demo：填寫表單送出)

Q2：資料散在哪裡？

LINE、紙本、Email、電話，找資料像考古。

用 Google Sheet 集中管理

以前是「找訊息」，現在是「查案件」。

(💡 現場 Demo：展示 Sheet 欄位自動匯入)

Q3：有沒有固定的判斷？

讓 GAS 抓關鍵字自動亮燈

「肯定、謝謝」 ➔ 🟢 綠燈
「提告、受傷」 ➔ 🔴 紅燈
「其他建議」   ➔ 🟡 黃燈
➔
if (有肯定內容) {
  綠燈;
} else if (有高風險) {
  紅燈;
}

(💡 概念版，非實際程式碼)

🤖

我不是手寫 GAS，我是這樣跟 AI 合作

🔄 我們的工作分配

🙋‍♂️ 我負責：講清楚流程
🤖 AI 負責：翻成 GAS 程式
🙋‍♂️ 我負責：貼上並測試結果
🤖 AI 負責：根據報錯修正錯誤

💬 我會這樣問 AI：

請幫我寫 Google Apps Script。

表單送出後，讀取機構欄位與意見內容。
如果內容有「提告、媒體、跌倒、受傷」，標成紅燈並寄信給主管、社工、書記。
如果有「肯定、謝謝」，標成綠燈並寄感謝信。
其他標成黃燈。

最後留下 ➔ 一套真的跑得動的工作流 🚀

Q4：發生事情時，要通知誰？

🟢 綠燈
主管
(收到好評)
🟡 黃燈
主管 + 社工 + 書記
(同步知悉)
🔴 紅燈
主管 + 社工 + 管理層
(危機通報)
👥 正副主管同步：
現場阿長有正副互搭與支援關係，相關主管會一起收到信，避免交接或支援時資訊斷掉。

「技術上是寄信，
管理上是把現場分工變成自動通知規則。」

⚙️ GAS 出場點：表單送出後，自動開始處理 (onFormSubmit)

Q5：寄出去以後，怎麼追蹤？

  • ⏳ 待處理 ➔ 🚧 處理中 ➔ ✅ 已結案
  • 自動封存：結案後系統自動收進倉庫。
  • 誤改救援：不小心改錯，還能搬回主表。
  • 手動變色：手動改燈號，自動換底色。
⚙️ GAS 出場點：表格修改後，自動觸發 (onEdit)

(💡 現場 Demo：手動改結案、觸發自動搬移)

Q6：主管最後想看什麼？

主管不想看 100 筆明細，只想知道：

紅燈幾件？待處理幾件？誰上榮譽榜？誰在警戒榜？

戰情室 ＋ 每天 17:00 自動日報

⚙️ GAS 出場點：時間到了，自動執行 (Time-driven)

(💡 現場 Demo：展示 Dashboard 與日報信件)

我們不是從程式開始，
而是從流程開始。

  • 📝 重複的事 ➔ 表單化
  • 🗂️ 分散的資料 ➔ 表格化
  • 🧠 固定的判斷 ➔ 規則化
  • 📨 固定的通知 ➔ 自動化
  • 📊 主管看結果 ➔ 報表化
🧠

所以，GAS 到底適合做什麼？

忘記程式碼沒關係，只要記得剛剛提到的三個「時機」：

  • 👉 表單送出時 (onFormSubmit)：自動分類、寄信
  • 👉 表格修改時 (onEdit)：改變顏色、結案搬移
  • 👉 時間到了時 (Time-driven)：每天 17:00 寄報表
